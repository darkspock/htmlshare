# htmlshare

AI-first publishing for generated HTML files.

`htmlshare` is not a website builder. It is a small file-sharing service for HTML/CSS/JS bundles generated by Claude, Codex, Cowork, or any other agent. The public home is plain HTML for SEO. The private product UI is React + Shadcn-style components. The backend is Go.

Product scope and roadmap decisions are tracked in [docs/product-spec.md](docs/product-spec.md).

## Run

```bash
go run ./cmd/htmlshare
```

The server listens on `http://localhost:4545`.

## Docker Compose

For an isolated local stack with PostgreSQL, MinIO, and Mailpit:

```bash
docker compose up -d --build
```

Published ports are intentionally non-standard to avoid collisions:

- App: `http://localhost:46545`
- PostgreSQL: `localhost:56432`
- MinIO API: `http://localhost:59100`
- MinIO console: `http://localhost:59101`
- Mailpit UI: `http://localhost:58025`
- Mailpit SMTP: `localhost:58026`

Compose credentials:

```text
PostgreSQL: htmlshare / htmlshare_dev_password / htmlshare
MinIO: htmlshare_minio / htmlshare_minio_password
Bucket: htmlshare-publications
```

The Compose app uses:

- `DATABASE_URL` for PostgreSQL-backed state.
- `S3_ENDPOINT`, `S3_BUCKET`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY` for MinIO-backed publication files.
- `SMTP_ADDR=mailpit:1025` for local email capture.
- `ABUSE_LLM_ENDPOINT` and `ABUSE_LLM_API_KEY` are optional. If omitted, abuse review uses a deterministic local heuristic.

Stop the stack:

```bash
docker compose down
```

## Production Deploy

Production deploys from GitHub Actions. The action builds the Docker image, pushes it to GHCR, then SSHes into the server and runs:

```bash
cd /opt/htmlshare
docker compose --env-file /opt/htmlshare/image.env -f /opt/htmlshare/docker-compose.yml pull app
docker compose --env-file /opt/htmlshare/image.env -f /opt/htmlshare/docker-compose.yml up -d --remove-orphans
```

The production compose file runs only the app container. It uses `network_mode: host` so the app can keep using the already-installed PostgreSQL, MinIO, Nginx, and email configuration from `/etc/htmlshare/htmlshare.env`.

Required GitHub secrets:

- `DEPLOY_HOST`: server host or IP.
- `DEPLOY_USER`: SSH user, usually `root`.
- `DEPLOY_SSH_KEY`: private SSH key for deployment.
- `DEPLOY_PORT`: optional, defaults to `22`.
- `GHCR_READ_TOKEN`: optional, only needed if the GHCR package is private.
- `GHCR_READ_USER`: optional, defaults to the GitHub actor.

## Fast AI Publish

An agent can publish a short-lived HTML bundle without email, OAuth, or an API key. It must generate a stable `agent_id` for the current conversation/session and reuse it.

```bash
curl -X POST http://localhost:4545/api/publish \
  -H "content-type: application/json" \
  -d '{
    "mode": "fast",
    "agent_id": "claude-session-unique-id",
    "agent_name": "claude",
    "title": "Quick File",
    "ttl_seconds": 86400,
    "files": {
      "index.html": "<!doctype html><html><body><h1>Quick File</h1></body></html>"
    }
  }'
```

Fast publications are public to anyone with the link and must expire within the configured maximum TTL.

## Registered Automation Signup

Use registered signup only when a human wants account ownership, analytics, recipients, domains, signed access, or longer-lived library control. An agent can start onboarding for that human with:

```bash
curl -X POST http://localhost:4545/api/ai/signup \
  -H "content-type: application/json" \
  -d '{
    "email": "owner@example.com",
    "name": "Owner",
    "agent": "claude",
    "intent": "account-owned-html-publishing"
  }'
```

The human receives a confirmation email. New unconfirmed registrations are provisional for 24 hours. If the email already belongs to a confirmed account, the API does not return a token to the agent; the account owner receives it by email or provides an existing `hsk_...` key.

## Simple Publish Protocol

Agents without MCP can publish account-owned files with one POST using either a session cookie or an API key from the dashboard:

```bash
curl -X POST http://localhost:4545/api/publish \
  -H "authorization: Bearer hsk_..." \
  -H "content-type: application/json" \
  -d '{
    "mode": "registered",
    "title": "Quarterly File",
    "visibility": "recipients",
    "require_registration": true,
    "files": {
      "index.html": "<!doctype html><html><head><link rel=\"stylesheet\" href=\"style.css\"></head><body><h1>Quarterly File</h1></body></html>",
      "style.css": "body{font-family:ui-serif,serif;margin:4rem}"
    },
    "share": {
      "emails": ["reader@example.com"],
      "message": "Here is the file."
    }
  }'
```

Visibility values:

- `private`: only the owner.
- `recipients`: invited emails or allowed domains.
- `public`: anyone with the link.

## Access Logs

Each request to a published file path under `/f/{slug}/...` creates an access log with:

- publication ID and slug
- requested path
- timestamp
- IP address from `X-Forwarded-For`, `X-Real-IP`, or `RemoteAddr`
- user agent
- user ID and email when the request is associated with a session
- allowed/denied and HTTP status

Owners can read stats with:

```bash
curl http://localhost:4545/api/library/{id-or-slug}/stats \
  -H "authorization: Bearer hsk_..."
```

The response includes `visits` and the raw `logs`.

## Signed Access and Comments

Owners can email a signed access token for legal proof:

```bash
curl -X POST http://localhost:4545/api/library/{id-or-slug}/signed-access \
  -H "authorization: Bearer hsk_..." \
  -H "content-type: application/json" \
  -d '{"email":"reader@example.com","message":"Please open this signed link."}'
```

Opening the emailed link confirms the reader email, records a signed access proof with IP, user agent, timestamp, and token ID, and creates a publication share for that reader.

Readers with access can comment on the whole page or on selected text. Published `index.html` pages receive a small injected widget that posts to:

```bash
POST /api/public-comments/{slug}
```

Comment payloads support `body`, `scope` (`general` or `inline`), `anchor_text`, `anchor_selector`, and `parent_id`. Owners can list, reply, archive, and soft-delete comments through:

```bash
GET  /api/library/{id-or-slug}/comments
POST /api/library/{id-or-slug}/comments
PATCH /api/comments/{comment-id}
DELETE /api/comments/{comment-id}
```

## Abuse Review

Anyone can submit an abuse notice:

```bash
curl -X POST http://localhost:4545/api/abuse-reports \
  -H "content-type: application/json" \
  -d '{"slug":"quarterly-file","reporter_email":"reader@example.com","reason":"phishing","details":"Asks for credentials."}'
```

The backend sends the case to `ABUSE_LLM_ENDPOINT` when configured, otherwise it uses the local heuristic. Blocking decisions immediately return `451` for the publication, create a strike, and critical cases or repeated strikes ban the owner email and publishing IP for 48 hours.

Run the MCP stdio server:


- `publish_html`
- `share_html`

The app UI lives in `web/app` and is React + Shadcn-style components:

```bash
cd web/app
npm install
npm run dev
```

The Go server serves `/` from `web/home/index.html` as direct static HTML.
